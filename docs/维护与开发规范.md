# 维护与开发规范

## 项目边界

- 后端入口：`cmd/pxe`。
- 服务编排：`internal/app`。
- 管理 API 和前端托管：`internal/web`。
- DHCP/ProxyDHCP：`internal/dhcp`。
- TFTP：`internal/tftp`。
- HTTP Boot：`internal/httpboot`。
- 通用网络地址工具：`internal/netutil`。
- 命令输出解码：`internal/command`。
- 启动菜单通用逻辑：`internal/bootmenu`。
- 启动 URL 拼接：`internal/booturl`。
- iPXE 脚本：`internal/ipxe`。
- 数据库和模型：`internal/storage`。
- 前端源码：`web/src`。
- 内置 iPXE 固件脚本：`embed.ipxe`。
- Docker 镜像构建流水线：`.github/workflows/docker.yml`。
- iPXE 固件构建流水线：`.github/workflows/build-boot.yml`。

## 修改规范

- 修改协议逻辑时同步更新日志和测试。
- 修改配置字段时同步更新默认值、校验、前端表单和文档。
- 修改 API 时保持统一响应结构，错误提示使用中文。
- API 响应中的列表字段必须返回空数组 `[]`，不能返回 `null`；Go 代码中列表结果使用 `make([]T, 0)` 或 `[]T{}` 初始化。
- 前端使用接口列表字段前必须做 `Array.isArray` 兜底，禁止直接假设后端数组一定非空非 `null` 后再读取 `.length`。
- 修改文件路径逻辑时必须保持根目录限制，禁止路径穿越。
- 文件内容接口只允许白名单文本类型和明确大小上限，禁止为了预览而读取镜像、固件、压缩包等大文件。
- 修改下发给客户端的 HTTP Boot 地址时必须复用 `internal/booturl`，避免端口、通告 IP 和监听地址处理不一致。
- 修改启动文件字段或架构识别时必须同步 `storage.BootFilesSettings`、`web/src/lib/types.ts`、服务配置页和 `internal/dhcp.bootFileCandidates`。
- 修改广播地址、定向广播或 WOL 目标推导时必须复用 `internal/netutil`。
- 修改命令执行输出展示时必须复用 `internal/command.DecodeOutput`，确保 Windows GBK/GB18030 输出不会乱码。
- 修改 UEFI/iPXE 菜单 timeout 时必须复用 `internal/bootmenu`，避免两条菜单链路行为漂移。
- 修改前端页面时保持现有 Tailwind/shadcn 风格，不引入另一套设计语言。
- 前端按钮默认必须保持横向文字，紧凑表格操作列优先用 `whitespace-nowrap`、固定最小宽度或图标按钮，避免中文按钮竖排。
- 修改 `embed.ipxe` 时必须同步使用文档和 `ai.md`，并确认 `build-boot.yml` 仍能使用 `EMBED=embed.ipxe` 构建 Legacy BIOS、UEFI x86_64、UEFI ARM64 产物。
- 修改 iPXE 固件构建流水线时必须保留 HTTPS 支持；公网 HTTPS 默认使用 iPXE 公共 CA/crosscert 机制，不要把 curl CA bundle 作为单个 `TRUST=` 文件注入。若改变产物名称，需要同步 README、使用文档和发布说明。
- 不提交运行时数据、数据库、日志、node_modules、dist 产物。

## 质量检查

提交前至少运行：

```bash
go test ./...
go vet ./...
npm run typecheck --prefix web
npm run build --prefix web
```

`npm run build --prefix web` 会直接输出到 `internal/web/dist`，Go 构建会把该目录嵌入二进制。
生产二进制使用 `go build -trimpath -ldflags="-s -w"`，避免携带本地路径、符号表和调试信息。

## 发布规范

- 通过 GitHub Actions 手动触发发布。
- 版本号使用 `vMAJOR.MINOR.PATCH`。
- Release 产物包含二进制和 README。
- 应用二进制发布使用 `.github/workflows/release.yml`；Docker 镜像发布使用 `.github/workflows/docker.yml`；iPXE 固件发布使用 `.github/workflows/build-boot.yml`，三者互不替代。
- iPXE 固件产物包括 `undionly.kpxe`、`ipxe-x86_64.efi`、`ipxe-arm64.efi`；这些是辅助启动文件，不是服务运行时自动生成文件。
- 发布前确认 `ai.md`、README 和 docs 与当前实现一致。

## 已知问题

- DHCP/ProxyDHCP 在不同固件、虚拟机和交换机下兼容性差异明显。
- 完整 DHCP 可能影响同网段真实设备，只能在明确网络边界后启用。
- TFTP 是 UDP 协议，丢包、MTU、blksize 都会影响稳定性。
- 外部下载能力必须提示来源 URL 和保存路径。
- 仪表盘和日志页必须共用 `web/src/lib/eventLog.ts`，避免重复 SSE、乱序刷新和无限日志堆积。
- 操作菜单保存后必须返回数据库中的最新操作列表，确保新建操作立即拿到 ID 并可执行。
- 操作菜单模板由后端按服务器平台返回，避免前端硬编码 Windows/Linux 不兼容命令。
