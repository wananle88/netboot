# 生产部署与编译

## 环境要求

- Go 1.25 或更高版本。
- Node.js 最新稳定版。
- npm。
- Windows 管理员权限或 Linux/macOS root 权限，用于绑定 DHCP/TFTP 等低端口。

## 本地生产构建

Windows：

```powershell
cd pxe
npm ci --prefix web
npm run build --prefix web
go test ./...
go vet ./...
New-Item -ItemType Directory -Force -Path dist | Out-Null
go build -trimpath -ldflags="-s -w" -o dist\pxe.exe .\cmd\pxe
```

Linux/macOS：

```bash
cd pxe
(cd web && npm ci && npm run build)
go test ./...
go vet ./...
mkdir -p dist
go build -trimpath -ldflags="-s -w" -o dist/pxe ./cmd/pxe
```

`-trimpath -ldflags="-s -w"` 是生产构建默认参数，用于去除本地路径、符号表和调试信息，减小二进制体积，降低首次启动时被系统安全扫描拖慢的概率。

## 交叉编译

前端构建会直接输出到 `internal/web/dist`，之后可交叉编译：

```powershell
$env:CGO_ENABLED="0"
$env:GOOS="linux"
$env:GOARCH="arm64"
New-Item -ItemType Directory -Force -Path dist | Out-Null
go build -trimpath -ldflags="-s -w" -o dist\pxe-linux-arm64 .\cmd\pxe
Remove-Item Env:\GOOS
Remove-Item Env:\GOARCH
Remove-Item Env:\CGO_ENABLED
```

纯 Go SQLite 使用 `modernc.org/sqlite`，不依赖 CGO，适合 Windows、Linux、macOS、OpenWrt、Armbian 等环境交叉编译。

## iPXE 固件构建

应用本体发布、Docker 镜像发布和 iPXE 固件构建是三条独立链路。`.github/workflows/build-boot.yml` 用于手动构建启动固件，不参与 `pxe` 服务二进制或 Docker 镜像的编译。

该流水线当前行为：

- 手动触发，可选输入 `tag_name`；非空时会把产物发布到对应 GitHub Release。
- 下载 iPXE v2.0.0 源码。
- 在 iPXE `general.h` 中开启 HTTPS 下载支持。
- 不传入 `TRUST=` 自定义 CA bundle，使用 iPXE 默认公共 CA/crosscert 信任机制；私有 CA 或完全离线 HTTPS 场景需要单独设计证书链。
- 将仓库根目录 `embed.ipxe` 复制进 iPXE 源码；文件不存在时才回退到最小 `dhcp` 脚本。
- 使用 `EMBED=embed.ipxe` 构建三个产物：`undionly.kpxe`、`ipxe-x86_64.efi`、`ipxe-arm64.efi`。
- 上传 `iPXE-Binaries` artifact，保留 3 天。

`embed.ipxe` 是固件内置脚本，当前菜单包含公网 Debian 12、Alpine Linux、iPXE Shell 和退出项，公网镜像源默认为清华 TUNA。修改该脚本后，需要重新运行 `build-boot.yml` 才会体现在固件产物里。

固件构建产物不会自动放入运行时 `data/boot/netboot`。生产环境需要按实际客户端架构放置或配置启动文件；例如 Legacy BIOS 常用 `undionly.kpxe`，UEFI x86_64 常用 `ipxe-x86_64.efi`，UEFI ARM64 常用 `ipxe-arm64.efi`。运行时服务配置已经提供 IA32/x64/ARM32/ARM64 独立 EFI 字段，混合架构网络中应保持文件名与客户端架构匹配。

## 生产运行建议

- ProxyDHCP 模式适合已有路由器 DHCP 的网络，是默认推荐模式。
- 完整 DHCP 模式只建议在隔离网络中使用，避免与路由器 DHCP 冲突。
- 管理端默认监听 `127.0.0.1:8088`，远程管理必须启用登录认证并限制访问来源。
- Windows 需要允许 UDP 67、UDP 69、UDP 4011 和 HTTP Boot 端口通过防火墙。
- Wake-on-LAN 需要网络允许 UDP 7/9 的广播或定向广播，并且目标设备 BIOS/UEFI 与网卡电源管理已启用网络唤醒。
- Linux 建议使用 root 运行，或为二进制授予绑定低端口所需 capability。
- 大镜像文件放在文件系统中，不放数据库。
- 操作菜单命令在服务器端执行，生产环境只应保存可信命令模板；命令不会经过 shell，但仍应限制管理端访问。

## Linux 一键部署脚本

仓库提供 `netboot.sh`，支持 apt、yum/dnf 和 Alpine apk 环境，服务管理支持 systemd 与 OpenRC。

```sh
sh netboot.sh
```

也可以直接使用子命令：

```sh
sh netboot.sh install
sh netboot.sh start
sh netboot.sh stop
sh netboot.sh update
sh netboot.sh uninstall
sh netboot.sh status
```

脚本会从 GitHub Releases 的 `latest/download` 地址下载当前架构对应的 Linux 包。默认安装到 `/usr/local/bin/pxe`，数据目录为 `/opt/netboot/data`，管理端监听 `0.0.0.0:8088`。安装时会同时下载 `undionly.kpxe`、`ipxe-x86_64.efi` 和 `ipxe-arm64.efi` 到 `/opt/netboot`，方便按需复制或在面板中配置使用。

## Docker 部署

项目提供轻量多阶段 `Dockerfile` 和 `docker-compose.yml`。Compose 默认使用 `network_mode: host`，便于 DHCP/ProxyDHCP、TFTP、HTTP Boot 和 WOL 直接使用宿主机网络。

当前功能只使用普通 TCP/UDP socket，不使用 ICMP、ARP 或 raw socket，因此不需要 `NET_RAW` 或 `NET_ADMIN`。容器通常只需要以 root 用户运行以绑定 UDP 67、UDP 69、UDP 4011 和 TCP 80 等低端口。

## GitHub 发布流水线

`.github/workflows/release.yml` 是手动触发流水线：

1. 输入版本号，例如 `v1.0.0`。
2. 执行前端类型检查、前端构建、Go 测试、`go vet`。
3. 构建 Windows、Linux、macOS 多平台二进制，包含 Windows/Linux ARM64 和 Linux ARMv7。
4. 打包为 zip/tar.gz。
5. 创建 GitHub Release 并上传产物。

`.github/workflows/build-boot.yml` 只构建 iPXE 启动固件，不构建 `pxe` 服务二进制。

`.github/workflows/docker.yml` 手动构建并推送 `linux/amd64`、`linux/arm64` 和 `linux/arm/v7` 镜像到 GHCR，标签为输入版本号和 `latest`。

Dependabot 每月检查 Go、npm 和 GitHub Actions 依赖更新。
